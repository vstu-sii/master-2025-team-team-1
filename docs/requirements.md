# Технические требования AI HR Assistant

## 1. Функциональные требования

### 1.1 Парсинг и обработка резюме

#### FR-1.1.1: Извлечение текста из резюме
- **Описание:** Система должна извлекать структурированный текст из резюме в различных форматах (PDF, DOCX, TXT, HTML).
- **Входные данные:** URL страницы с резюме на поддерживаемых портальных сайтах (hh.ru, Habr Career, LinkedIn).
- **Выходные данные:** JSON-структура с полями: ФИО, контакты, опыт работы, образование, навыки, достижения, языки.
- **Приемлемость:** Точность извлечения ≥ 85% для структурированных полей.

#### FR-1.1.2: Определение типа страницы
- **Описание:** Расширение должно распознавать, открыта ли страница с резюме или вакансией.
- **Логика:** Проверка структуры DOM, URL-паттернов, наличия ключевых элементов.
- **Ошибка:** Если страница не распознана, вывести пользователю уведомление с предложением повторить.

#### FR-1.1.3: Обработка массовых резюме
- **Описание:** Возможность обработки списков кандидатов (10–100+ резюме за сессию).
- **Очередь обработки:** Асинхронная обработка с очередью (queue), чтобы не перегружать систему.
- **Кэширование:** Сохранение спаршенных данных на клиенте (IndexedDB) на 7 дней.

---

### 1.2 Анализ соответствия и скоринг

#### FR-1.2.1: Анализ соответствия вакансии
- **Входные данные:** Спаршенное резюме и описание вакансии.
- **Процесс:** Сравнение ключевых параметров (опыт, навыки, уровень, язык программирования и т.д.).
- **Выходные данные:** Скоринг соответствия (0–100%), категория кандидата (Top/Medium/Low).
- **Метрика точности:** Валидация на корпусе 500+ резюме с ручной разметкой.

#### FR-1.2.2: Выявление навыков и пробелов
- **Описание:** Система должна выделять:
  - **Сильные стороны:** навыки, которые совпадают или превышают требования.
  - **Пробелы:** недостающие или слабые навыки.
  - **Риски:** расхождения, которые могут повлиять на кандидата.
- **Выходные данные:** Структурированный объект с массивами strengths, gaps, risks.
- **Пример:** 
  ```json
  {
    "strengths": ["5 лет Python", "опыт с AWS"],
    "gaps": ["нет опыта с Docker", "junior уровень DevOps"],
    "risks": ["большой перерыв в карьере (1 год)"]
  }
  ```

#### FR-1.2.3: Ранжирование кандидатов
- **Описание:** На основе скоринга система ранжирует кандидатов в порядке убывания релевантности.
- **Выходные данные:** Отсортированный список с визуализацией (Top 10%, Medium 50%, Low 40%).
- **Стабильность:** Ранжирование должно быть воспроизводимым при одних и тех же входных данных.

---

### 1.3 Генерация вопросов для интервью

#### FR-1.3.1: Генерация персонализированных вопросов
- **Входные данные:** Профиль кандидата (резюме), требования вакансии, выявленные пробелы.
- **Выходные данные:** 5–7 вопросов, категоризированных по типам:
  - Технические вопросы (для проверки hard skills).
  - Поведенческие вопросы (для оценки soft skills и культуры).
  - Опыт-ориентированные вопросы (на основе реальных проектов кандидата).
- **Требование:** Каждый вопрос должен быть уникален и релевантен пробелам кандидата.

#### FR-1.3.2: Регенерация вопросов
- **Описание:** Возможность пользователя перегенерировать вопросы с другим фокусом (более технический, поведенческий и т.д.).
- **Параметры:** Focus type (technical, behavioral, experience, role-specific).

---

### 1.4 Экспорт и интеграция

#### FR-1.4.1: Экспорт данных
- **Форматы:** CSV, Excel (XLSX), JSON.
- **Содержимое:** ФИО, скоринг, сильные стороны, пробелы, рекомендованные вопросы.
- **Сохранение:** На локальное устройство или облачное хранилище (Google Drive, Telegram).

#### FR-1.4.2: Интеграция с внешними сервисами
- **Поддерживаемые сервисы:** Google Drive, Telegram Bot (для отправки уведомлений).
- **Авторизация:** OAuth 2.0 для Google, Token-based для Telegram.

---

### 1.5 Авторизация и безопасность

#### FR-1.5.1: Авторизация пользователя
- **Метод:** OAuth 2.0 / JWT токены.
- **Область действия:** Пользователь должен быть авторизован перед использованием всех функций.
- **Механизм:** Refresh tokens (сроком на 30 дней), access tokens (15 минут).

#### FR-1.5.2: Управление доступом (RBAC)
- **Роли:** Admin (полный доступ), HR Manager (анализ, экспорт), Viewer (только просмотр).
- **Область:** Отдельные вакансии, проекты, группы кандидатов могут быть приватными/общими.

#### FR-1.5.3: Защита данных
- **Шифрование:** Все данные резюме передаются по HTTPS, хранятся зашифрованными в БД.
- **GDPR:** Соответствие EU GDPR для данных граждан ЕС (право на удаление, экспорт).
- **Логирование:** Все действия пользователя логируются для аудита.

## 2. Нефункциональные требования

### 2.1 Производительность

#### NFR-2.1.1: Время отклика на парсинг резюме
- **Требование:** Парсинг одного резюме ≤ 2 секунды (с кэшем ≤ 500 мс).
- **Сценарий:** 20 резюме обработано за ≤ 10 минут (батч-обработка).

#### NFR-2.1.2: Время генерации анализа и вопросов
- **Требование:** Анализ и генерация вопросов ≤ 3 секунды (за счет оптимизации промптов и кэширования).
- **Система:** Асинхронное выполнение на backend с WebSocket/Server-Sent Events (SSE) для real-time обновлений.

#### NFR-2.1.3: Пропускная способность
- **Требование:** Система должна обрабатывать ≥ 100 одновременных пользователей без деградации.
- **Масштабирование:** Горизонтальное масштабирование backend'а (контейнеры, load balancer).

#### NFR-2.1.4: Время загрузки расширения
- **Требование:** Расширение должно инициализироваться за ≤ 1 секунду после загрузки страницы.

---

### 2.2 Надежность и доступность

#### NFR-2.2.1: Коэффициент доступности
- **Требование:** SLA ≥ 99.5% (максимум 3.6 часа downtime в месяц).
- **Механизм:** Репликация БД, резервные узлы, failover автоматизация.

#### NFR-2.2.2: Восстановление после сбоев
- **Требование:** RTO ≤ 15 минут (время восстановления), RPO ≤ 5 минут (потеря данных).
- **Бэкап:** Ежедневное резервное копирование, геораспределённые реплики.

#### NFR-2.2.3: Обработка ошибок
- **Требование:** При ошибке парсинга система должна уведомить пользователя и предложить альтернативные действия.
- **Пример:** Если резюме не распознано, предложить загрузить вручную или повторить попытку.

---

### 2.3 Масштабируемость

#### NFR-2.3.1: Рост объема данных
- **Требование:** Система должна поддерживать растущий объем резюме (от 1000 до 10 млн за 18 месяцев).
- **Архитектура:** Партиционирование БД по user_id, индексирование ключевых полей, денормализация для быстрых запросов.

#### NFR-2.3.2: Кэширование
- **Требование:** Использовать Redis для кэширования часто запрашиваемых данных (анализы, вопросы).
- **TTL:** 24 часа для анализов, 7 дней для спаршенных резюме.

---

### 2.4 Удобство использования (UX)

#### NFR-2.4.1: Адаптивная верстка
- **Требование:** Интерфейс должен работать на всех устройствах (desktop, tablet, mobile).
- **Breakpoints:** 320px, 768px, 1024px, 1440px.

#### NFR-2.4.2: Доступность (Accessibility)
- **Требование:** Соответствие WCAG 2.1 уровень AA.
- **Элементы:** ARIA-метки, клавиатурная навигация, контрастность текста.

#### NFR-2.4.3: Поддержка браузеров
- **Требование:** Поддержка Chrome, Firefox, Safari, Edge последних двух версий.
- **Обратная совместимость:** ES2020+, CSS Grid, Flexbox.

---

### 2.5 Безопасность и конфиденциальность

#### NFR-2.5.1: Защита от атак
- **Требование:** OWASP Top 10 mitigations (SQL injection, XSS, CSRF, brute-force).
- **Механизм:** Input validation, parameterized queries, Content Security Policy (CSP).

#### NFR-2.5.2: Конфиденциальность данных
- **Требование:** Резюме не передаются третьим сторонам без явного согласия пользователя.
- **Соглашение:** Явное согласие при первом входе + опция "Do Not Track".

## 3. Метрики производительности

### 3.1 Ключевые показатели эффективности (KPI)

| Метрика | Целевое значение | Способ измерения |
|---------|-----------------|------------------|
| **Время на первичный отбор** | ≤ 10 мин / 20 резюме | Время session'а пользователя |
| **Точность скоринга** | ≥ 85% | A/B тестирование с ручной разметкой |
| **Adoption rate** | ≥ 60% в месяц | Кол-во активных пользователей / зарегистрировано |
| **Retention rate (30-day)** | ≥ 40% | Возврат пользователя за 30 дней |
| **NPS (Net Promoter Score)** | ≥ 50 | Ежемесячный опрос пользователей |

### 3.2 Технические метрики (SLI/SLO)

| Метрика | SLO | SLI (инструмент) |
|---------|-----|------------------|
| **Latency (p99)** | ≤ 3 сек | Prometheus + Grafana |
| **Error Rate** | ≤ 0.1% | Application logs + ELK Stack |
| **Availability** | ≥ 99.5% | Uptime monitoring (Datadog/New Relic) |
| **Throughput** | ≥ 100 req/sec | Load testing (Apache JMeter / Locust) |
| **Hallucination Rate** | ≤ 5% | Manual review + Langfuse |
| **Token Usage** | ≤ 2000 tokens/request | LLM API monitoring |

### 3.3 Мониторинг

#### Инструменты:
- **Application Performance Monitoring (APM):** Langfuse для LLM, Datadog для приложения.
- **Infrastructure Monitoring:** Prometheus + Grafana для метрик сервера.
- **Logs:** ELK Stack (Elasticsearch, Logstash, Kibana) для централизованного логирования.
- **Алерты:** PagerDuty для критических инцидентов.

## 4. Архитектурные принципы

### 4.1 Архитектурный стиль

#### API-First Architecture
- Основной интерфейс взаимодействия — REST API / GraphQL.
- Фронтенд и браузерное расширение — клиенты API.
- Все бизнес-логика на backend.

#### Микросервисная архитектура (при масштабировании)
- **Сервис парсинга:** Ответственен за извлечение текста из резюме.
- **Сервис анализа:** Работает с LLM, вычисляет скоринг.
- **Сервис генерации:** Создает вопросы и отчеты.
- **Сервис авторизации:** Управление токенами, RBAC.
- Общая шина сообщений: RabbitMQ / Apache Kafka для async communication.

#### Event-Driven Architecture
- Все значимые события логируются (resume_parsed, analysis_completed, export_requested).
- Event handlers обновляют кэши, отправляют уведомления, логируют аналитику.

---

### 4.2 Технологический стек

#### Backend
- **Framework:** FastAPI (async, высокая производительность, автодокументирование).
- **Language:** Python 3.11+.
- **БД:** PostgreSQL (основные данные) + Redis (кэш).
- **LLM Integration:** LangChain / LlamaIndex для работы с моделями (OpenAI, Anthropic, локальные).
- **Task Queue:** Celery + Redis для асинхронных задач.

#### Frontend
- **Framework:** Next.js 14+ (SSR, SEO-friendly, performance).
- **Language:** TypeScript.
- **State Management:** Redux Toolkit или Zustand.
- **UI Components:** Shadcn/ui или Material-UI.
- **Styling:** Tailwind CSS.

#### Browser Extension
- **Manifest:** Chrome Extension Manifest V3.
- **Content Script:** Инъекция в DOM страницы резюме.
- **Background Service Worker:** Асинхронная обработка, коммуникация с backend.
- **Popup UI:** React компоненты в iframe.

#### Infrastructure
- **Containerization:** Docker + Docker Compose.
- **Orchestration:** Kubernetes (для production).
- **CI/CD:** GitHub Actions (тесты, сборка, deploy).
- **Monitoring:** Prometheus + Grafana + Loki.

### 4.3 Дизайн данных

#### Entity Relationship Model

```
Users
├── id (PK)
├── email
├── name
├── role (admin, hr_manager, viewer)
└── created_at

Vacancies
├── id (PK)
├── user_id (FK → Users)
├── title
├── description
├── required_skills (JSONB array)
└── created_at

Resumes
├── id (PK)
├── user_id (FK → Users)
├── vacancy_id (FK → Vacancies)
├── raw_text
├── parsed_data (JSONB: name, experience, skills, education)
├── created_at

Analyses
├── id (PK)
├── resume_id (FK → Resumes)
├── vacancy_id (FK → Vacancies)
├── score (0-100)
├── strengths (JSONB array)
├── gaps (JSONB array)
├── risks (JSONB array)
├── questions (JSONB array)
└── created_at

Exports
├── id (PK)
├── user_id (FK → Users)
├── format (csv, xlsx, json)
├── data (JSONB)
├── exported_at
```

#### Кэширование (Redis)
```
Key: analysis:{resume_id}:{vacancy_id}
Value: JSON (score, strengths, gaps, risks, questions)
TTL: 86400 (24 часа)
```

## 5. Ограничения и допущения

### 5.1 Ограничения

- **Языки:** На начальном этапе поддержка только русского и английского языков.
- **Форматы резюме:** PDF (до 20 МБ), DOCX (до 10 МБ), TXT (до 5 МБ).
- **Портальные сайты:** Поддержка hh.ru, Habr Career, LinkedIn на начальном этапе.
- **Параллельная обработка:** Максимум 5 резюме одновременно на одного пользователя (очередь).

### 5.2 Допущения

- Пользователи имеют стабильное интернет-соединение.
- Резюме содержат достаточно информации для анализа (минимум 100 слов).
- Вакансия описана на русском или английском языке.
- Система не является юридическим советником и не гарантирует качество найма.
