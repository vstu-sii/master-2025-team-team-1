# Требования к ML-моделям для проекта «AI HR Assistant"



---

## Краткое резюме (TL;DR)

- **Архитектура:** гибрид — эмбеддинги для поиска и ранжирования + LLM для генерации и объяснений; оркестрация через LangChain.
- **Latency (E2E для интерактивного запроса):** целевой диапазон **≤ 20–30 секунд** (анализ одного резюме на странице + генерация 5–7 вопросов).
- **Throughput:** целевая пропускная способность **≥ 500 резюме/день** (MVP: ≥ 200/день).
- **Качество:** точность соответствия вакансии ≥ **85%** (в пилотной валидации с HR); генерация вопросов — автоматические метрики (BLEU/ROUGE/BERTScore) + human review.
- **Приватность:** не хранить персональные данные дольше, чем необходимо; предусмотреть возможность on-prem deployment для корпоративных клиентов.

---

## 1. Цели моделей и их роли

1. **Семантический поиск и ранжирование** — эмбеддинги для преобразования резюме и вакансий в векторы и поиска по схожести.
2. **Структурирование резюме (NER, parsing)** — извлечение навыков, опыта, образования, контактов.
3. **Скоринг соответствия** — модель/классификатор, дающий интерпретируемый скор соответствия вакансии.
4. **Генерация персонализированных вопросов** — LLM генерирует 5–7 релевантных вопросов, нацеленных на обнаруженные пробелы в навыках.
5. **Explainability** — краткое пояснение «почему кандидат в шорт-листе» (1–3 аргумента).

---

## 2. Выбор типов моделей — рекомендации

### 2.1. Модели эмбеддингов (рекомендация)

- **Sentence-Transformers (например, all-mpnet-base-v2)** — основной выбор для качественных эмбеддингов и точного семантического поиска.
- **all-MiniLM-L6-v2** — лёгкий вариант для быстрого первичного фильтра (меньше latency, дешевле).

**Применение:** индексация резюме, быстрый retrieval, предпросеивание кандидатов перед LLM.

### 2.2. LLM для генерации и объяснений (рекомендация)

Стратегия «двух уровней»:

1. **Лёгкие/эффективные LLM (on‑prem или self-hosted)** — для большинства интерактивных задач, где важен низкий latency и контроль над данными (пример: модели ~7B параметров).
2. **Мощные LLM (cloud / API)** — для сложных анализов, финальной генерации и повышенной связности ответов (использовать выборочно для hard-cases).

**Практический подход:** использовать лёгкую модель для day-to-day задач и вызывать крупную LLM для финальной проверки/увеличенной глубины обработки.

---

## 3. Требования к производительности (SLA)

- **E2E latency (интерактив):** медиана ≤ 20 с, p95 ≤ 30 с для сценария: извлечение текста с веб-страницы → эмбеддинг → ранжирование → генерация 5–7 вопросов.
- **Throughput (бекенд):** обеспечить обработку ≥ 500 резюме/день. Для этого реализовать batch-режим, очереди и кэширование эмбеддингов.
- **Хардварные ориентиры:** эмбеддинговые модели — inference на CPU; LLM ~7B — ≈12–16 GB VRAM для inference; большие модели 40B+ — десятки десятков GB VRAM (требуют специализированных GPU).

## 5. Метрики качества и валидация

### Для matching / ranking
- **Accuracy, Precision, Recall, F1** — для бинарной классификации "подходит / не подходит".
- **NDCG@k, MAP** — оценка качества ранжирования.
- **Цель:** accuracy ≥ 85% на контрольной выборке, при сохранении баланса precision/recall.

### Для генерации вопросов
- **Автоматические метрики:** BLEU, ROUGE, BERTScore — для быстрой автоматической проверки.
- **Human eval:** 3–5 HR-специалистов оценивают релевантность и полезность вопросов по шкале 1–5; целевой средний балл ≥ 4.0.

### Производительность
- **Latency metrics:** median и p95 E2E. Цели: median ≤ 20 с, p95 ≤ 30 с.
- **Throughput metrics:** requests/hour, processed resumes/day.

### План валидации
1. Собрать контрольный набор: 500–1,000 резюме + 50–100 вакансий с разметкой HR.
2. Бенчмарки: сравнить комбинации эмбеддингов и LLM по метрикам выше.
3. Human-in-the-loop: регулярная проверка с получением обратной связи от HR.
4. A/B тестирование в продакшне и мониторинг drift.

## 6. Приватность, соответствие нормативам и стоимость

### Приватность и соответствие
- Резюме содержат PII — обязательно шифрование in-transit и at-rest.
- Для облачных вызовов LLM: заключение DPA, гарантии обработки/неиспользования данных для обучения (по требованию клиента).
- Предусмотреть возможность on-prem deploy или VPC-isolated deployment для корпоративных клиентов.

### Стоимость
- Коммерческие API (cloud LLM) при больших объёмах могут быть дорогими. Как mitigation — гибридный сценарий: lightweight self-hosted модели + cloud LLM для сложных случаев.

## 7. Архитектурный пайплайн (пример реализации)

1. **Browser Extension (Client):**
   - Извлечение текста с веб-страницы / OCR для PDF/изображений.
   - Отправка minimal context на бекенд + показываем прогресс.

2. **Backend API:**
   - *Preprocessing:* нормализация, language detection, NER.
   - *Embedding & Retrieval:* генерация эмбеддингов → ANN search (FAISS / Milvus / Pinecone).
   - *Ranking / Scoring:* скор соответствия вакансии.
   - *LLM Generation:* генерация персонализированных вопросов и короткого обоснования.
   - *Caching & Feedback:* кэш эмеддингов и результатов; HR-метки используются для будущего ретренинга.

3. **Orchestration:**
   - LangChain (или кастомный orchestrator) для паттерна RAG: retrieve → augment → generate.

---

## 8. Эксперименты и бенчмарки (план)

1. **Эмбеддинги:** сравнить all-mpnet-base-v2 vs all-MiniLM-L6-v2 на NDCG@10 и latency для набора из 1,000 резюме.
2. **LLM-генерация:** Mistral-7B vs альтернативы (локальные и cloud) — оценка BLEU/ROUGE и blind human eval.
3. **Интеграционный тест:** симуляция 500 резюме/день — замер costs, p95 latency.
4. **Privacy test:** проверка redaction pipeline и соответствия требованиям клиентов.

---

## 9. Rollout-план

- **PoC (1–2 недели):** all-MiniLM + Mistral-7B / GPT-J — проверка latency и качества.
- **MVP (4–6 недель):** hybrid deployment: embeddings on CPU, LLM inference on GPU (managed cloud), закрытый запуск с группой HR.
- **Scale:** добавить более мощные LLM или cloud provider для финальных задач; оптимизации quantization, autoscaling.
- **Enterprise:** on-prem вариант + DPA + SLA.
