# Требования к ML-моделям AI HR Assistant (LLM-сервис)

Этот документ описывает требования к моделям, которые используются в сервисе `llm-service`
(оценка резюме, генерация вопросов, структурирование резюме, эмбеддинги, NER).

## 1. Цели моделей

Основные функциональные задачи:

1. **Оценка соответствия резюме вакансии**
   - Вход: текст резюме + текст описания ваканссии.
   - Выход: числовой скор соответствия (0–100) + краткое текстовое объяснение.

2. **Генерация вопросов для интервью**
   - Вход: текст резюме и/или описания ваканссии.
   - Выход: список уточняющих вопросов для собеседования.

3. **Структурирование резюме**
   - Вход: неструктурированный текст резюме.
   - Выход: JSON-структура с полями (ФИО, контакты, опыт, образование, навыки).

4. **Резюме-карточка кандидата**
   - Вход: текст резюме.
   - Выход: краткое summary (2–3 предложения) для рекрутера.

5. **Эмбеддинги и similarity**
   - Вход: текст (резюме, вакансия, фрагмент).
   - Выход: вектор фиксированной длины + метрики близости (cosine similarity).

---

## 2. Типы моделей

### 2.1. LLM (основная модель)

Задачи:

- генерация текста (объяснения, вопросы);
- извлечение структуры (JSON) из неструктурированного текста;
- оценка соответствия резюме вакансии (через промпт).

Требования:

- Поддержка **русского и английского** языков.
- Контекст минимум **1024–2048 токенов** (достаточно для 1 резюме и 1 описания вакансии).
- Возможность управлять:
  - `max_new_tokens` (длина ответа),
  - `temperature` (степень креативности),
  - seed/детерминизм (по возможности).

Примеры моделей (baseline):

- `google/flan-t5-base` или аналогичная open-source Seq2Seq модель (через `transformers`).
- При необходимости — более крупные модели, совместимые с API.

### 2.2. Модель эмбеддингов

Задачи:

- представление текстов в векторном виде;
- вычисление семантической близости резюме и вакансий;
- baseline-оценка similarity для benchmarking.

Требования:

- Sentence-level эмбеддинги.
- Русский и английский языки.
- Размер вектора: не менее 256, предпочтительно 512–1024.

Пример baseline:

- `sentence-transformers/paraphrase-multilingual-mpnet-base-v2`.

## 3. Требования к производительности

### 3.1. Latency

Цель для baseline-сервиса:

- Время ответа LLM-эндпоинтов (оценка, вопросы, структурирование) — **до 3–5 секунд** на запрос.
- Время получения эмбеддингов — **до 300 мс** на один текст средней длины.

### 3.2. Throughput

- Поддержка параллельных запросов: 5–10 RPS при горизонтальном масштабировании.
- Возможность батчевой обработки (на уровне эмбеддингов) для ускорения оффлайн-оценки.

---

## 4. Метрики качества

### 4.1. Оценка соответствия резюме

Если есть размеченные `ground_truth_score`:

- **MAE (Mean Absolute Error)** между оценкой модели и разметкой.
- **MSE (Mean Squared Error)**.
- **R² (коэффициент детерминации)**.

Дополнительно:

- Корреляция (Spearman/Pearson) между модельными и ручными оценками.

### 4.2. Генерация вопросов для интервью

На основе `ground_truth_questions`:

- **ROUGE-1 / ROUGE-2 / ROUGE-L** между сгенерированными и эталонными списками вопросов.
- Ручная оценка:
  - релевантность вопросам вакансии и резюме;
  - конкретика;
  - отсутствие дубликатов.

### 4.3. Структурирование резюме (JSON)

Метрики:

- Доля **валидных JSON-ответов** (успешный `json.loads` и соответствие ожидаемой схеме).
- Доля **полностью/частично заполненных** ключевых полей:
  - `candidate_name`,
  - `contact_info.email`,
  - хотя бы один пункт `experience` и `education`.

### 4.4. Эмбеддинги и similarity

При наличии пар текстов с `ground_truth_similarity`:

- MAE/MSE/R² по сравнению с модельной similarity (cosine similarity).
- Корреляция с ручными оценками.

## 5. Ограничения

### 5.1. Память и вычислительные ресурсы

- Минимальный таргет среды выполнения:
  - CPU-машина 4–8 vCPU, 8–16 GB RAM;
  - опционально GPU для ускорения LLM и эмбеддингов.
- LLM baseline должен работать на CPU (на учебных/студенческих ресурсах),
  более тяжёлые модели — опционально, через GPU или внешние API.

### 5.2. Стоимость

- Предпочтительно использование **open-source моделей** и локального инференса.
- При использовании внешних API:
  - фиксировать лимиты по количеству токенов;
  - логировать количество запросов и объём трафика.

### 5.3. Приватность и безопасность

- Резюме и вакансии могут содержать PII (ФИО, контакты).
- Требования:
  - хранить данные только на время обработки и экспериментов;
  - при логировании — маскировать PII (email, телефон, ссылки);
  - при использовании внешних LLM-API явно указывать в документации, что данные уходят третьей стороне
    (для учебного проекта — только с искусственно сгенерированными или анонимизированными резюме).

---

## 6. Нефункциональные требования

- **Повторяемость**:
  - фиксированные версии моделей (через `requirements.txt`);
  - фиксированные параметры генерации (temperature, max_new_tokens).
- **Логирование и наблюдаемость**:
  - логирование запросов/ответов (с маскированием PII);
  - возможная интеграция с Langfuse для трейсинга запросов к LLM.
- **Документация**:
  - описать контракт API (`ml/api`),
  - описать формат данных (`ml/evaluation/test_samples.json`),
  - описать метрики и сценарии использования (`ml/reports/baseline_report.md`).

---

## 7. План развития

- Перейти на более мощные модели (Qwen, Gemma, Mistral) при наличии ресурсов.
- Добавить эмбеддинги и similarity как отдельный слой.
- Внедрить полноценный offline-benchmark с ручной разметкой и метриками MAE/MSE/корреляции.
- Интегрировать Langfuse для продвинутого мониторинга запросов к LLM.
