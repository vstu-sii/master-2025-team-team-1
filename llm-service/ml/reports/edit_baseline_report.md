# Baseline-модель для оценки соответствия резюме вакансии

## 1. Цель

Цель baseline-модели — дать простую, но воспроизводимую оценку того,
насколько резюме кандидата соответствует конкретной вакансии.

Формулировка задачи:

- **Вход**:
  - текст описания ваканссии (`job_description`),
  - текст резюме (`resume`).
- **Выход**:
  - числовой скор соответствия на шкале **0–100**,
  - краткое текстовое объяснение (на русском), описывающее, почему выставлен такой балл.

Данный baseline используется:

- как демонстрация работоспособного ML/LLM-пайплайна в спринте 2;
- как отправная точка для дальнейших улучшений (эмбеддинги, более мощные LLM, прод-API и полноценный evaluation).

---

## 2. Архитектура baseline-решения

Baseline-модель реализована как гибридный подход:

1. **LLM-компонента** (инференс через `transformers`):
   - модель: `google/flan-t5-base` (открытая, подходит для локального инференса и Colab);
   - роль модели: «HR-специалист в IT», который читает вакансию и резюме и объясняет оценку.

2. **Эвристический baseline по ключевым словам**:
   - если LLM не возвращает корректный ответ (нет числа, пустой вывод и т.п.),
   - включается fallback: вычисляется доля пересечения ключевых технологий и содержательных терминов между вакансией и резюме,
   - результат переводится в скор 0–100.

Таким образом, система всегда возвращает числовую оценку и объяснение, даже если LLM ведёт себя нестабильно (что типично для небольших моделей без дообучения).

---

## 3. Реализация прототипа (Colab + сервис)

### 3.1. LLM-инференс

В прототипе используется:

- `AutoTokenizer` и `AutoModelForSeq2SeqLM` из `transformers`;
- базовая модель `google/flan-t5-base` (для Colab — также тестировался `flan-t5-small`).

Промпт-инжиниринг:

- модель получает роль HR-специалиста;
- на вход подаются:
  - текст ваканссии,
  - текст резюме;
- от модели ожидается ответ в формате:

- строка с префиксом `SCORE:` и целым числом от 0 до 100;
- строка с префиксом `EXPLANATION:` и кратким текстовым объяснением на русском.

### 3.2. Парсинг ответа LLM

Ответ парсится регулярными выражениями:

- извлекается число после `SCORE:`;
- извлекается текст после `EXPLANATION:`.

Если формат нарушен (нет числа, нет нужных префиксов), модельный ответ считается некорректным, и включается эвристический baseline.

### 3.3. Эвристический baseline

Если:

- LLM возвращает пустую строку,  
  **или**
- ответ не содержит `SCORE:` и корректного числа,

используется **эвристический скор**:

1. Текст вакансии и резюме токенизируется (слова → `lower-case`, без пунктуации).
2. Убираются стоп-слова (RU + EN).
3. Выделяются:
   - общий набор «meaningful» токенов;
   - подмножество тех-токенов `TECH_KEYWORDS`:
     `python`, `django`, `fastapi`, `rest`, `api`, `postgresql`, `docker`, `git`,
     `sql`, `pandas`, `numpy`, `sklearn` и т.п.
4. Считаются два оверлапа:

   - по технологиям:

     `skill_overlap = |TECH(vacancy) ∩ TECH(resume)| / |TECH(vacancy)|`

   - по всем смысловым токенам:

     `token_overlap = |TOKENS(vacancy) ∩ TOKENS(resume)| / |TOKENS(vacancy)|`

5. Финальный скор:

`score = 100 * (0.7 * skill_overlap + 0.3 * token_overlap)`

Explanation в этом случае формируется как:

> Оценка рассчитана автоматически на основе совпадения ключевых технологий  
> и содержательных терминов между резюме и описанием ваканссии.  
> Чем больше общих технологий (Python, фреймворки, БД, Docker и т.п.), тем выше балл.

## 4. Тестовый набор кейсов (мини-датасет)

Для демонстрации качества baseline-модели подготовлен небольшой синтетический набор из 5 кейсов:

1. **Сильное совпадение: Python backend**

   - **Вакансия**: Python backend-разработчик (REST API, PostgreSQL, FastAPI/Django, Docker, Git).
   - **Резюме**: 3 года Python backend, REST API, FastAPI/Django, PostgreSQL, Docker, Git, тесты, code review.
   - **Ожидание**: высокий скор (*high_match*).

2. **Среднее совпадение: Python + немного другого стека**

   - **Вакансия**: Python в финтех-проекте (Python + веб-фреймворк + БД + очереди как плюс).
   - **Резюме**: Python-скрипты, немного Flask, учебный опыт с БД, базовый Docker/Git, без очередей.
   - **Ожидание**: средний скор (*medium_match*).

3. **Слабое совпадение: фронтенд под backend-вакансию**

   - **Вакансия**: Python backend (REST API, PostgreSQL).
   - **Резюме**: frontend-разработчик (JS/TS/React), почти без Python и БД.
   - **Ожидание**: низкий скор (*low_match*).

4. **Почти не совпадает: графический дизайнер**

   - **Вакансия**: data engineer / backend (Python, SQL, ETL, Airflow/Kafka/Spark).
   - **Резюме**: графический дизайнер (Figma, Photoshop, Illustrator).
   - **Ожидание**: очень низкий скор (*low_match*).

5. **Другая доменная область: Data Scientist**

   - **Вакансия**: Data Scientist (Python, pandas, sklearn, ML-метрики).
   - **Резюме**: Data Scientist 2 года (Python, pandas, sklearn, ROC-AUC, F1, SQL).
   - **Ожидание**: высокий скор (*high_match*).

Результаты для этих кейсов собираются в таблицу (`pandas.DataFrame`) с полями:

- `case_name`,
- `score`,
- `bucket` (`high_match`, `medium_match`, `low_match`),
- `explanation`,
- `raw_llm_output` (сырой ответ LLM для анализа поведения).

## 5. Наблюдения и анализ

### 5.1. Поведение LLM (`flan-t5-base`)

Модель иногда:

- повторяет куски инструкции (например, `"No JSON, no additional text..."`);
- возвращает очень короткий фрагмент текста;
- не соблюдает строгий формат `SCORE / EXPLANATION`.

Поэтому для надёжности используется **гибридный подход: LLM + эвристика**.

### 5.2. Качество эвристики

Эвристика по ключевым словам:

- хорошо различает сильное, среднее и слабое совпадение;
- интуитивно понятна (больше общих технологий → выше балл).

Ограничения:

- не учитывает важность отдельных технологий;
- не смотрит на контекст и порядок слов;
- чувствительна к качеству текста (опечатки, разный регистр и т.п.).

### 5.3. Hallucinations

В текущей реализации текст `Explanation` от LLM/эвристики:

- не используется напрямую для принятия решений,
- служит только как человекочитаемое пояснение.

Для production-уровня потребуется:

- более строгая валидация утверждений в объяснении по исходному тексту;
- возможно, отдельный модуль `"justification checker"`.

## 6. Интеграция с сервисом

Baseline-инференс интегрирован в сервисную архитектуру:

- `ml/models/baseline.py` — клиент для локального API (LLM + эмбеддинги);
- `ml/api/main.py` — FastAPI-приложение с эндпоинтами:
  - `POST /generate-text`
  - `POST /evaluate-suitability`
  - `POST /generate-interview-questions`
  - `POST /structure-resume`
  - `POST /get-embedding`
  - `POST /extract-entities`

Это позволяет:

- вызывать baseline-модель из других компонентов;
- логировать запросы/ответы;
- в будущем подключить **Langfuse** для трейсинга.

---

## 7. Ограничения и план развития

### 7.1. Ограничения

- Модель `flan-t5-base` без дообучения, нестабильный формат ответа.
- Тестирование на небольшом синтетическом датасете.
- Эвристика не использует эмбеддинги и семантическую similarity.

### 7.2. План улучшений

**Модели:**

- перейти на более мощную LLM (Qwen / Gemma / Mistral) при наличии ресурсов;
- добавить слой эмбеддингов для similarity (`sentence-transformers`).

**Качество:**

- собрать реальный датасет (вакансия–резюме–оценка HR);
- посчитать MAE/MSE/R² и корреляцию с ручной оценкой;
- построить ROC-кривые для классификации «подходит / не подходит».

**Инженерия:**

- вынести эвристику в отдельный модуль;
- настроить Langfuse для мониторинга и отладки LLM-вызовов;
- добавить unit-тесты на парсинг ответов LLM и работу fallback-а.

---

## 8. Вывод

В рамках спринта 2 реализован рабочий baseline-прототип оценки соответствия резюме вакансии:

- показан LLM-инференс через `transformers` и промпт-инжиниринг;
- добавлен устойчивый эвристический baseline по ключевым словам;
- подготовлен набор тестовых кейсов и проведена качественная оценка результатов;
- всё это интегрировано в архитектуру сервиса (модели, API, отчёт).

Этот baseline не является финальным решением, но демонстрирует полный ML-пайплайн
и создаёт основу для последующих улучшений.
